Active threads over time
This graph represents the real number of virtual users injected during the test. This graph must be as close as possible to the teorethical injection schedule defined in the test plan. If the real test plan does not closely follow the original it means that the injector had problems because the number of virtual users was too high.

Aggregated response times
These are the response times aggregated in columns. The response time bars should be aggregated in a single group. If there are different groups of columns separated by a significative response time it means that some of the responses took significantly more time to be processed that the others during the test duration. It is good to have all the columns aggregated to the left of the graph.

Transactions per second during the test
The number of transactions per second should closely follow the amount of virtual users performing requests during the time taken by the test to run. A deviation from the curve usually means problems in the system under load, that is not able to follow the injection schedule.

Estimated transactions per second by virtual users
This graph represents an estimation of the maximum number of transactions per second that the server is able to process depending of the number of enabled virtual users. This graph should represent a line. Ideally, for a server not in the stress zone, twice the threads should lead to twice the transactions per second. A system starts to have problems when the line starts to oscillate. The important values in this graph are those with a huge number of samples. This number of users corresponds to the test steps. For example for a step of 10 virtual users, the most important values will be 10 and 20 users. The red point represents the average of users and transactions per second during the test.

Load injector: CPU
The injector CPU should always be below 100% usage. This graph represents the total CPU usage adding the CPU usage of all available cores during the test.

Load injector: Memory
The injector memory should always be below 100% usage.

Load injector: SWAP
The injector should never make use of the swap. Using the swap menas that there were not enough RAM. If there are swap usage consider adding more RAM to the injector.

Load injector: Network usage
This graph displays the incoming and outcoming network data usage during the test. Neither the incoming nor the outcoming values should have a clear ceiling. If a clear ceiling appears it means that the injector network card acted as a bottleneck and either the load should be reduced or a more capable network connectioin should be added to the injector.
